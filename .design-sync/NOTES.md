# Design-sync notes — Facturia UI

## Repo context
- This is an **application** (not a standalone design system package). The "design system" is Facturia's own shadcn/ui copies living in `front-end/src/components/ui/`, customised with a custom indigo/oklch colour palette.
- Package manager: **pnpm** (pnpm-lock.yaml in `front-end/`)
- CSS: **TailwindCSS v4** (plugin-based, `@import "tailwindcss"` in `index.css` — processed by `@tailwindcss/vite`). No `tailwind.config.js`.
- Font: **Inter Variable** via `@fontsource-variable/inter` (npm package, bundled via Vite)
- Path alias: `@/` → `front-end/src/`

## CSS build approach
TailwindCSS v4 can only be compiled via the Vite plugin. `buildCmd` runs the full Vite app build (`pnpm -C front-end build`) then copies the hashed-filename CSS output to a stable path `front-end/dist/facturia-ui.css`. The `cssEntry` points at this stable path.

Re-sync: when component source changes, re-run `buildCmd` to refresh the CSS.
After each rebuild the hashed output is copied automatically by the build command.

## Synced components
**20 components** — all from `front-end/src/components/`:
- `ui/`: AlertDialog, Avatar, Badge, Button, Card, DropdownMenu, Input, Label, ScrollArea, Select, Separator, Sheet, Sidebar, Skeleton, Sonner, Tabs, Textarea, Tooltip
- Root-level custom: BrandName, ModeToggle

**Excluded** (app-specific, deep context deps): AppSidebar, ChatMessage, TemplateGenerating, FieldError, NavSidebar, PresetSheet, TemplateCard, TemplateThumbnail

## Known transitive deps
- `@/lib/utils` (`cn()` helper — clsx + tailwind-merge) → bundled via synth-entry
- `@/hooks/use-mobile` (used by Sidebar) → bundled via synth-entry
- All Radix UI primitives in `front-end/node_modules/`
- `class-variance-authority`, `clsx`, `tailwind-merge`, `next-themes`, `lucide-react`, `sonner`

## Provider setup (if needed)
- **ModeToggle** uses `useTheme()` from `next-themes` — needs `ThemeProvider` wrapper if previews fail with "useTheme must be used inside ThemeProvider"
- **Sidebar** uses `useSidebar()` context — needs `SidebarProvider` wrapper if standalone Sidebar sub-components fail
- If provider errors appear: set `cfg.provider` to `ThemeProvider` (from next-themes) with `attribute: "class"`

## Known render warns
- `[RENDER_BLANK]` Button (floor card only) — floor card renders blank because Button requires visible children to show content. Non-blocking; resolved by authored `previews/Button.tsx`. Safe to ignore on re-sync if Button preview still exists.

## Re-sync commands
```bash
# Rebuild authored previews (after editing .design-sync/previews/<Name>.tsx):
node .ds-sync/lib/preview-rebuild.mjs --config .design-sync/config.json \
  --node-modules front-end/node_modules --out ds-bundle \
  --components Badge,Button,Card,...

# Capture screenshots for grading:
node .ds-sync/package-capture.mjs --out ds-bundle --components Badge,Button,Card,...
# Note: package-capture.mjs ignores --config/--node-modules (pass only --out and --components)

# CSS rebuild (when component source changes):
pnpm -C front-end build && node -e "const fs=require('fs');const f=fs.readdirSync('front-end/dist/assets').find(n=>n.endsWith('.css'));if(f)fs.copyFileSync('front-end/dist/assets/'+f,'front-end/dist/facturia-ui.css');else process.exit(1)"
```

## Re-sync risks
- `cssEntry` path (`front-end/dist/facturia-ui.css`) is generated by the build command; if `buildCmd` fails or the copy step fails, cssEntry will be stale
- Tailwind CSS purges unused classes — if the app stops using a class that a component needs, it'll disappear from the compiled CSS
- `vite/client` type is in tsconfig.app.json — if it causes esbuild issues, switch to `tsconfig.json`
